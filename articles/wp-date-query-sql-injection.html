What Is WP_Date_Query?

WP_Date_Query is a WordPress core helper class introduced to allow developers to filter post and content queries by date. It is used internally by WP_Query and is widely used in plugins and themes to retrieve posts within a given date range — for example, "show all posts published this month" or "find events between two dates."

Because it generates raw SQL WHERE subclauses and is frequently exposed to user-controlled input, it has historically been a target for SQL injection. In October 2022, security researcher Michael Mazzolini disclosed a critical flaw in WP_Date_Query that affected every WordPress installation from version 3.7 up to and including 6.0.2.

Affected Versions: WordPress 3.7 through 6.0.2.
Fixed In: WordPress 6.0.3 (and backported to all supported branches: 5.9.5, 5.8.6, 5.7.8, 5.6.10, 5.5.11, 5.4.12, 5.3.14, 5.2.17, 5.1.15, 5.0.18, 4.9.22, 4.8.21, 4.7.25, 4.6.25, 4.5.28, 4.4.29, 4.3.30, 4.2.34, 4.1.37, 4.0.37, 3.9.38, 3.8.40, 3.7.40), released October 17, 2022.
Source: WordPress 6.0.3 Security Release – WordPress.org

The Vulnerability: Improper Sanitization of the relation Parameter

The root cause was a missing validation step in the sanitize_query() method of WP_Date_Query. When building compound date conditions, the class accepts a relation key that controls how multiple date clauses are joined in SQL — either AND or OR. Before the fix, the code only checked whether the value was explicitly 'OR'; any other string was silently passed through as 'AND' without being stripped of SQL metacharacters.

According to the Wordfence threat intelligence advisory, the vulnerability was classified as CWE-89 (Improper Neutralization of Special Elements used in an SQL Command) with a Wordfence CVSS score of 8.8 (High). WPScan assigned CVSS 4.0 and Patchstack assigned CVSS 3.7 — the difference reflects each organization's assessment of exploitability under real-world conditions.

The Vulnerable Code Path

The following simplified reconstruction shows what the vulnerable sanitize_query() looked like before the patch. The key problem is that $relation was only tested for the value 'OR'; any other value — including payloads with injected SQL — was accepted without further validation: 

// VULNERABLE – class-wp-date-query.php (WordPress < 6.0.3)
public function sanitize_query( $queries ) {
    foreach ( $queries as $key => $query ) {
        if ( 'relation' === $key ) {
            // Only checks for 'OR' — anything else passes through untouched
            $queries['relation'] = strtoupper( $query ) === 'OR' ? 'OR' : 'AND';
        }
        // ...
    }
    return $queries;
}

Because the comparison used strtoupper( $query ) and the result was just defaulted to 'AND', the relation value itself was never written into the final SQL. However, in certain execution paths where the raw value was used before this check ran — or where plugins used the value directly without going through the sanitizer — malicious SQL fragments could reach the query.

Exploitation Context

The vulnerability is most dangerous when a plugin or theme passes user-controlled input directly into the date_query argument of WP_Query without first validating the relation key. A simplified exploit would look like this: 

// DANGEROUS – Plugin code that passes raw user input into date_query
$args = [
    'post_type'  => 'event',
    'date_query' => [
        'relation' => sanitize_text_field( $_GET['rel'] ), // insufficient!
        [
            'after' => '2024-01-01',
        ],
    ],
];
$query = new WP_Query( $args );
// If $_GET['rel'] = "AND (SELECT 1 FROM (SELECT SLEEP(5))a)-- "
// and sanitize_text_field() does not strip SQL metacharacters,
// the injected fragment can reach the final SQL statement.
Important context: sanitize_text_field() removes HTML tags but does not protect against SQL injection. Always use the database-level defenses described below.

The Fix: A Dedicated sanitize_relation() Method

The patch, committed to the WordPress develop repository as commit d815d2e, introduced a new private method that acts as a strict allowlist: 

// FIXED – class-wp-date-query.php (WordPress >= 6.0.3)
public function sanitize_relation( $relation ) {
    if ( 'OR' === strtoupper( $relation ) ) {
        return 'OR';
    } else {
        return 'AND';
    }
}

This looks similar to the old logic but the critical difference is that sanitize_relation() is now called consistently every time the relation value is used to build SQL, not just during initial parsing. The return value is always either the literal string 'OR' or the literal string 'AND' — no other value can reach the query.

The Patchstack security release summary confirms: "WP does not properly sanitize the relation passed to WP_Date_Query, which could lead to SQL injection" and credits the discovery to Michael Mazzolini.

How WP_Date_Query Builds SQL

Understanding the internals helps you write safer plugin code. When you pass a date_query argument to WP_Query, WordPress instantiates WP_Date_Query and calls get_sql(), which assembles the WHERE clause fragment: 

// What WP_Date_Query generates for a compound date query
$args = [
    'date_query' => [
        'relation' => 'OR',
        [
            'after'  => 'January 1st, 2024',
            'before' => 'December 31st, 2024',
        ],
        [
            'after' => 'January 1st, 2026',
        ],
    ],
];

// Resulting SQL fragment (simplified):
// WHERE (
//   (post_date > '2024-01-01' AND post_date < '2024-12-31')
//   OR
//   (post_date > '2026-01-01')
// )

The relation value ('OR' above) is placed verbatim between the subclauses. This is exactly why unchecked values are dangerous — anything written there ends up directly in the SQL string.

Safe Usage of WP_Date_Query in Plugin Development

1. Allowlist the relation Parameter

Even on WordPress 6.0.3+, defense in depth means you should validate input yourself before it ever reaches core: 

// SAFE – validate relation before passing to WP_Query
function get_events_by_date( $relation = 'AND' ) {
    // Strict allowlist — only 'AND' or 'OR' are valid
    $relation = strtoupper( $relation );
    if ( ! in_array( $relation, [ 'AND', 'OR' ], true ) ) {
        $relation = 'AND';
    }

    $args = [
        'post_type'  => 'event',
        'date_query' => [
            'relation' => $relation,
            [
                'after'     => '1 month ago',
                'inclusive' => true,
            ],
        ],
    ];

    return new WP_Query( $args );
}

2. Never Pass Raw $_GET / $_POST Values

// DANGEROUS – raw user input directly in date_query
$args = [
    'date_query' => [
        'relation' => $_GET['relation'],   // never do this
        'column'   => $_GET['column'],     // never do this either
        [
            'year' => $_GET['year'],       // needs intval()
        ],
    ],
];

// SAFE – sanitize every user-supplied value individually
$allowed_relations = [ 'AND', 'OR' ];
$relation = strtoupper( sanitize_key( $_GET['relation'] ?? 'AND' ) );
if ( ! in_array( $relation, $allowed_relations, true ) ) {
    $relation = 'AND';
}

$year = absint( $_GET['year'] ?? date( 'Y' ) );

$args = [
    'date_query' => [
        'relation' => $relation,
        [
            'year' => $year,
        ],
    ],
];

3. Validate the column Parameter

The column key in WP_Date_Query specifies which database column to filter on. Accepting this value from user input is equally risky because column names are not quoted by $wpdb->prepare(): 

// SAFE – allowlist valid date columns
$allowed_columns = [
    'post_date',
    'post_date_gmt',
    'post_modified',
    'post_modified_gmt',
];

$column = sanitize_key( $_GET['column'] ?? 'post_date' );
if ( ! in_array( $column, $allowed_columns, true ) ) {
    $column = 'post_date';
}

$args = [
    'date_query' => [
        [
            'column' => $column,
            'after'  => '2024-01-01',
        ],
    ],
];

4. Use $wpdb->prepare() for Custom Date Queries

When WP_Date_Query cannot cover your use case and you need to write raw SQL, always use prepared statements. According to the official WP_Date_Query class reference, the class is designed as a helper, not a replacement for good parameterized query practices: 

// SAFE – raw date query using $wpdb->prepare()
global $wpdb;

$start_date = sanitize_text_field( $_GET['start'] ?? '' );
$end_date   = sanitize_text_field( $_GET['end']   ?? '' );

// Validate that inputs are actual dates before using them
if ( ! strtotime( $start_date ) || ! strtotime( $end_date ) ) {
    return [];
}

$results = $wpdb->get_results(
    $wpdb->prepare(
        "SELECT ID, post_title FROM $wpdb->posts
         WHERE post_type   = %s
           AND post_status = %s
           AND post_date BETWEEN %s AND %s
         ORDER BY post_date DESC",
        'event',
        'publish',
        $start_date,
        $end_date
    )
);

Checking Whether Your Site Is Patched

The simplest check is your WordPress version. Log in to your dashboard and go to Dashboard → Updates. If you are running WordPress 6.0.3 or later you have the fix. According to the WPScan vulnerability record, all branches back to 3.7 received a backported patch — there is no supported version of WordPress where this remains unpatched.

If you maintain a plugin that uses date_query arguments, audit every place where user-supplied data feeds into the relation or column keys. Even on a fully patched WordPress, your plugin can reintroduce the vulnerability if it passes unsanitized values.

Lessons for Plugin and Theme Developers

  • Trust nothing from the browser: $_GET, $_POST, and $_REQUEST are attacker-controlled. Validate every key and value against an explicit allowlist before using it in any query argument.
  • Allowlists beat blocklists: Trying to remove dangerous characters from a string is fragile. Checking that the value is exactly 'AND' or 'OR' is unbreakable.
  • sanitize_text_field() is not SQL-safe: It strips HTML and extra whitespace but does not prevent SQL injection. Use it for display-safe strings; use allowlists or $wpdb->prepare() for database safety.
  • Defense in depth: Even after WordPress patched core, a plugin that pre-processes and re-validates the same data would have been safe regardless. Don't rely solely on the framework catching your mistakes.
  • Keep WordPress and plugins updated: This vulnerability existed in core from version 3.7 (released 2013) until 6.0.3 (2022) — almost a decade. Sites that stayed up to date received the patch the day it was released.

Conclusion

The WP_Date_Query SQL injection vulnerability is a textbook example of how a small missing validation — one sanitizer method not called consistently enough — can create a high-severity attack surface in a codebase used by hundreds of millions of websites. The fix itself was equally small: a strict allowlist that allows only 'AND' or 'OR' to reach the final SQL string.

If you build plugins or themes that accept date range inputs from users, apply the same principle: validate the relation and column parameters against an explicit allowlist, use absint() for integer date components, and fall back to $wpdb->prepare() whenever WP_Date_Query is not sufficient. Patch your WordPress installation if you have not already — this is a zero-effort, high-impact step that closes the vulnerability entirely.
