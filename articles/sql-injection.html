Understanding SQL Injection Attacks

SQL Injection is one of the most dangerous and common web vulnerabilities, consistently ranking in the OWASP Top 10. It occurs when an attacker inserts malicious SQL code into input fields, allowing them to manipulate database queries. In WordPress sites, this can lead to data theft, unauthorized access, or complete database compromise. Understanding how to prevent SQL injection is critical for protecting your WordPress installation and user data.

Critical Warning: SQL Injection can allow attackers to read sensitive data, modify database contents, execute administrative operations, and even gain system-level access. Always use prepared statements and never trust user input.

How SQL Injection Works

Classic SQL Injection Example

Consider a simple login form that checks credentials against a database. A vulnerable query might look like this: 

// VULNERABLE - WordPress example
$username = $_POST['username'];
$password = $_POST['password'];

// DANGEROUS: Direct string concatenation
$query = "SELECT * FROM wp_users WHERE user_login = '$username' AND user_pass = '$password'";
$result = $wpdb->query($query);

An attacker could enter ' OR '1'='1 as the username, causing the query to become: 

SELECT * FROM wp_users WHERE user_login = '' OR '1'='1' AND user_pass = ''

This bypasses authentication entirely because '1'='1' is always true.

Advanced SQL Injection Techniques

  • Union-based: Extract data by combining results with legitimate queries
  • Time-based blind: Infer data through response delays
  • Boolean-based blind: Determine truth values through conditional responses
  • Stacked queries: Execute multiple SQL statements
  • Out-of-band: Exfiltrate data through DNS or HTTP requests

WordPress-Specific Prevention

Use WordPress Prepared Statements

WordPress's $wpdb object provides the prepare() method for safe database queries. Always use placeholders: 

// SAFE - Using $wpdb->prepare()
global $wpdb;

$username = $_POST['username'];
$password = $_POST['password'];

// Use %s for strings, %d for integers, %f for floats
$query = $wpdb->prepare(
    "SELECT * FROM $wpdb->users WHERE user_login = %s AND user_pass = %s",
    $username,
    $password
);

$result = $wpdb->get_results($query);

Use WordPress Query Methods

WordPress provides helper methods that automatically sanitize queries: 

// Safe query using WordPress methods
global $wpdb;

$post_id = $_GET['id'];

// get_results() with placeholders
$results = $wpdb->get_results(
    $wpdb->prepare(
        "SELECT * FROM $wpdb->posts WHERE ID = %d",
        $post_id
    )
);

// get_var() for single values
$count = $wpdb->get_var(
    $wpdb->prepare(
        "SELECT COUNT(*) FROM $wpdb->posts WHERE post_author = %d",
        get_current_user_id()
    )
);

// get_row() for single row
$post = $wpdb->get_row(
    $wpdb->prepare(
        "SELECT * FROM $wpdb->posts WHERE post_name = %s",
        $post_slug
    )
);
Best Practice: Always use $wpdb->prepare() for any query containing user input. This is the gold standard in WordPress security.

Placeholder Types in WordPress

// Different placeholder types
%d  // Integer
%f  // Float
%s  // String

// Example with mixed types
$results = $wpdb->get_results(
    $wpdb->prepare(
        "SELECT * FROM $wpdb->posts 
         WHERE post_author = %d 
         AND post_title LIKE %s 
         AND post_date > %s",
        $author_id,
        '%' . $wpdb->esc_like($search) . '%',
        $date
    )
);

Escaping Output from Queries

After retrieving data from the database, always escape it before displaying: 

// Retrieve and escape
$user_data = $wpdb->get_row(
    $wpdb->prepare(
        "SELECT * FROM $wpdb->users WHERE ID = %d",
        $user_id
    )
);

// Safe output
echo esc_html($user_data->user_login);
echo esc_attr($user_data->user_email);
echo esc_url($user_data->user_url);

Avoiding Custom Table Prefixes

Always use the $wpdb prefix variable to handle table names: 

// WRONG - Hardcoded table name
$results = $wpdb->get_results("SELECT * FROM wp_posts WHERE ID = 1");

// CORRECT - Uses dynamic prefix
$results = $wpdb->get_results(
    "SELECT * FROM $wpdb->posts WHERE ID = 1"
);

WordPress Plugin Development Security

Securing Custom Post Type Queries

// Plugin: Custom post type with secure queries
class SecurityPlugin {
    public function get_custom_posts($author_id) {
        global $wpdb;
        
        // Validate input
        $author_id = absint($author_id);
        if ($author_id <= 0) {
            return [];
        }
        
        // Safe query
        return $wpdb->get_results(
            $wpdb->prepare(
                "SELECT ID, post_title, post_excerpt 
                 FROM $wpdb->posts 
                 WHERE post_author = %d 
                 AND post_type = %s 
                 AND post_status = %s",
                $author_id,
                'security_article',
                'publish'
            )
        );
    }
}

Handling Search Queries Safely

// Safe search implementation
function secure_search($search_term) {
    global $wpdb;
    
    // Sanitize search input
    $search_term = sanitize_text_field($search_term);
    
    // Limit search term length
    if (strlen($search_term) > 100) {
        return [];
    }
    
    // Use LIKE with prepared statement
    // esc_like() escapes wildcards
    return $wpdb->get_results(
        $wpdb->prepare(
            "SELECT * FROM $wpdb->posts 
             WHERE (post_title LIKE %s OR post_content LIKE %s)
             AND post_status = %s",
            '%' . $wpdb->esc_like($search_term) . '%',
            '%' . $wpdb->esc_like($search_term) . '%',
            'publish'
        )
    );
}

Testing for SQL Injection

Common Test Payloads

// Basic authentication bypass
' OR '1'='1
' OR 1=1 --
admin' --
' OR 'a'='a

// UNION-based extraction
' UNION SELECT NULL, NULL, NULL --
' UNION SELECT user_login, user_pass, user_email FROM wp_users --

// Time-based blind
' AND SLEEP(5) --
' AND IF(1=1, SLEEP(5), 0) --

// Boolean-based blind
' AND '1'='1
' AND '1'='2

Testing Checklist

  • Test all form inputs (search, login, filters, etc.)
  • Test URL parameters (?id=1, ?page=1)
  • Test custom post type queries
  • Test plugin custom fields
  • Test contact form submissions
  • Use automated scanners like SQLmap (with permission)

Common Mistakes to Avoid

  • Direct string concatenation: Never build queries by concatenating user input
  • Over-reliance on escaping: Escaping alone is not enough; use prepared statements
  • Forgetting to validate input types: Always check if input is an integer when expecting one
  • Storing passwords in plaintext: Use WordPress's wp_hash_password()
  • Trusting user input: Never assume user input is safe
  • Error messages leaking information: Don't display raw database errors to users

Conclusion

SQL Injection remains one of the most critical web vulnerabilities. In WordPress, always use $wpdb->prepare() for any query containing user input. For static sites with database access, use your framework's parameterized query system. Combine prepared statements with input validation and output escaping for defense-in-depth protection.

Security is a mindset. Make it a habit to always validate input, use prepared statements, and regularly test your code for vulnerabilities. Your users' data depends on it.
